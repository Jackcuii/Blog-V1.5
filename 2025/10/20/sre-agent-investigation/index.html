Cleric

What they claim

Cleric is diagnosing recurring alerts with supporting evidence. Investigations land in Slack: clear, structured, and actionable. Feedback helps it get smarter, quicker, and more context-aware”
Workflow:

  1. Starts with alert
  2. builds a hypothesis tree
  3. test potential causes against logs, metrics, and traces.
  4. Delivers the most likely root cause, a confidence score, and a step-by-step reasoning trail.
  5. Sends a clear diagnosis with supporting evidence to Slack

Materials & Key points

  1. No info in the Doc…
  2. feature Transparency (Shows every step of the investigation)
  3. they only want to make a helper into the human workflow
  4. workflow really like Datadog’s
  5. They claim: Mapping infrastructure and dependencies with minimal setup, keeping this context current as systems evolve
  6. Ecosystem: They claim: Works with Datadog, PagerDuty, AWS, GCP, Kubernetes(including control-plane), and more without extra config. Handles AWS, GCP, Azure, and hybrid stacks. Show outputs in slack.
  7. Trigger: (traditional) alert hits from your observability tools, cloud provider, or CI/CD pipeline
  8. Noise Cancellation: They claim they have, no details.
  9. Continual Learning & Memorization: They claim they have, no details.

Additional info

They claims they have samples on:

  • Alert: Investigation: Application error spikes, High latency, Distributed system failures, Failed deployments, Database issues, Network bottlenecks, Ingress and gateway errors, Kubernetes workload crashes, Container OOMKills, Queue processing delays
  • RCA: Silent degradations, Memory leaks, Scaling bottlenecks, Certificate expiry, Resource exhaustion (Kubernetes), Cluster health issues (Kubernetes), Kubernetes config errors, Controller misconfigurations
  • Fix recommendation: Database and backend issues, Job and CronJob failures (Kubernetes), Container crashes and OOMKills, Kubernetes configuration errors, Certificate expiry
    And some Case study: How the World’s Leading Community-Based Travel Network is Transforming Incident Response with AI

Azure SRE Agent (Preview)

What they claim

Diagnose, mitigate, and resolve incidents across Azure Monitor or integrated tools. The agent works autonomously or with approvals.
RCA, Mitigation
Workflow:

  1. receives an alert from the management platform
  2. brings the incident into its context
  3. analyzes the incident, reviews logs, health probes, and other telemetry
  4. summarizes findings, determines if the alert is a false positive, and decides whether action is needed.

Materials & Key points

  1. Their auto-mitigation is limited to pre-defined workflow
  2. Ecosystem: work within the Azure ecosystem.
  3. Trigger: By human prompt / the alert from: - Azure Monitor alerts, PagerDuty, ServiceNow
  4. Noise Cancellation: Not mentioned.
  5. Continual Learning & Memorization: Not mentioned.

Additional info

They mentioned general mitigation in the future.
They also mentioned normal operations of the cloud here

resolve.ai

What they claim

Resolve AI is helping engineering teams reimagine work with a multi-agent AI SRE that triages alerts, investigates complex issues, and operates autonomously in the most demanding environments across clouds and regions.
Detection, RCA, Mitigation
Workflow:

  1. receive alert
  2. Transforms scattered documentation, telemetry, and observability data into context.
  3. Formulates plans, tests hypotheses, surfaces root causes with evidence, and explains outcomes.
  4. Uses production tools to propose or execute changes,
  5. Build long-last memory to accumulate experience

Materials & Key points

  1. The key feature of their effort is using the knowledge graph to model the cluster.
  2. Ecosystem:
  3. Trigger: not clear, seems to be By alert

Datalog Bits AI

What they claim

Bits AI is a suite of intelligent AI capabilities that work across your critical monitoring, development, and security workflows. Bits can autonomously investigate alerts, suggest code fixes, review security signals, and take action on external systems—all with full contextual, organizational, and operational awareness. Empower your SREs, developers, and security teams to operate at scale, zero-in on the highest priority issues, and resolve incidents faster.
Detection, RCA, Mitigation
Workflow:

  1. Autonomously triages alerts
  2. use logs, traces, metrics, Real User Monitoring (RUM) events, security signals, and runtime variables in real time, allowing it to build a rich production context
  3. do root cause investigations using agentic reasoning and complete access to Datadog telemetry and service knowledge
    (make hypotheses, then invalidates hypotheses without supporting evidence and digs deeper into promising leads. Each hypothesis is classified as validated, invalidated, or inconclusive (tree))
  4. deliver real-time summaries, next-step suggestions, and auto-generated postmortems.
    video: https://imgix.datadoghq.com/img/blog/bits-ai-sre/Investigation.mov?fm=webm&fit=max

Materials & Key points

  1. The mitigation mostly for fix the bug in the production codes (“Generates production ready code changes, complete with tests and can create pull requests, to accelerate issue resolution without compromising code quality”)
  2. They also use the tree structure to check the RCA hypo (like Cleric) It is natural since it is just like structured prompt engineering
  3. Ecosystem: Whole Datadog ecosystem.
  4. Trigger: By alert / Human prompt(They have an interface to dialog with the AI to ask AI to do something, so also support normal ops)
    Similar to Azure SRE Agent
  5. Noise Cancellation: Not mentioned.
  6. Continual Learning & Memorization: They claim they have, no details。
    (“It draws on memory from past alerts to recognize patterns and accelerate investigations. If the issue has occurred before, Bits remembers what worked and what didn’t. You can correct Bits when it makes a mistake or reinforce the right diagnosis when it gets it right. “)

Additional info
